Style Guide for TDS Developments in Python¶

This style guide describes the recommended practices for TDS developments in Python at SAES.

1. Naming Conventions¶

  • Variables and Functions:

    • Use “snake_case” conventions: my_function, local_variable.

  • Classes:

    • Use “CamelCase” style: ClassName, MyTestClass.

  • Constants:

    • Use uppercase with underscores: GLOBAL_CONSTANT, MAX_VALUE.

2. Documentation¶

  • Docstrings: Each public function, class, and module must include a docstring using triple quotes """.

    • Use Numpy style docstrings.

    • Include purpose, description, and usage in the docstring.

    • Parameters: Specify units if applicable.

    • Notes:

      • Author: Name of the contributor.

      • DDA (if exists): Relevant .docx file and link (stored in docs/DDAs).

      • References (if exists): Standards or related documentation.

      • Test Files (if exists): Include input and output file paths for result verification, unless the test is clearly defined in the Parallel Code and Test Structure. E.g .. code-block:

        def signal_spectrum_calculation(data):
"""
Calculates the spectrum of a signal.
Parameters
----------
data : ndarray
  Input signal.
Returns
-------
ndarray
  The frequency spectrum of the signal.

Examples (Optional)
--------
>>> x = np.arange(0,100)
>>> y = signal_spectrum_calculation(x)
>>> plt.plot(y)

Notes
-----
Author: Pablo Aguirre
DDA: DDA - Descripción de Algoritmos_Simulador_FLS_A0.docx
Refs: STANAG 1454
Test files:
in: tests/data/data_test_signal_spectrum_calculation
out: tests/data/data_test_signal_spectrum_calculation_result
"""

  • Docs:

    • Repositories should have a clear and concise readme.md to help navigate the documentation and give users an idea of the package contents.

    • Repositories should have a “docs” folder where specific algorithm documentation is stored.

3. Parallel Code and Test Structure¶

To ensure a systematic testing approach, we adhere to a parallel directory structure between the source code (src) and the test files (tests). This setup facilitates straightforward mapping between source files and their respective tests.

/your_project
│
├── /src
│   └── /package
│       ├── __init__.py
│       └── core.py          # Contains functions (e.g., def add_numbers)
│
└── /tests
    ├── __init__.py
    └── test_core.py         # Contains tests (e.g., class TestAddNumbers for add_numbers)

Key Points:

  • File Correspondence: Every .py file in src has a matching test_*.py file in tests.

  • Function-Class Mapping: Each function in core.py must have a dedicated test class in test_core.py (e.g., add_numbers is tested by class TestAddNumbers).

This structure ensures that each component of your library is adequately tested and that there is clear traceability between code and tests.

4. Writing Clean and Readable Code¶

  • Small Functions: Functions should do one thing well and be size-limited to facilitate readability.

  • Clear Comments: Add comments where the code is not self-explanatory.

  • Avoid Redundancy: Reuse existing functions and modules instead of duplicating logic.

5. Use of Tools¶

  • Linting: Use ruff check to maintain code quality.

  • Formatting: Adopt ruff format for consistent code formatting.

  • Testing: Use pytest to write and execute unit tests.

  • Automation with Tox: Utilize tox to automate and manage linting, formatting, and testing processes across multiple environments.

6. Version control¶

  • Versioning: Adopt semantic versioning 2.0.0.

  • Git: Perform small and descriptive commits, use branches for new features.

7. Management of Large Files¶

Git is not a good tool for storing large files.¶

In fact, it’s not ideal for storing any files that are not plain text (PDFs, Word, etc.). However, as long as these files are small (<5/10 MB), they can be directly uploaded to Git. For larger datasets, especially those that may change over time, it is strongly recommended NOT to upload them to Git. To manage these datasets, it is better to reference their location on OneDrive or similar, and ideally, create an automatic script to download the data and place it in the desired folder.

8. Code Formatting¶

  • Indentation: Use 4 spaces per indentation level. Do not use tabs.

  • Line Length: Lines should not exceed 99 characters.

  • Whitespace:

    • Add two blank lines between functions and classes.

    • Avoid trailing whitespace at the end of lines.

9. Error and Exception Handling¶

  • Use specific exceptions whenever possible and avoid general exceptions. Handle exceptions properly: .. code-block:

    try:
    result = critical_function()
except ValueError as e:
    print(f"Value error: {e}")

10. Import Organization¶

  • Import standard Python modules first, followed by third-party modules, and finally local project imports with a blank line between each group, for example: .. code-block:

    import os
import sys
import numpy as np
import scipy.signal
from tds_project.module import specific_function

11. Import Aliases¶

  • Don’t use import aliases except from:

    • Approved Aliases:

      • import numpy as np

      • import pandas as pd

      • import matplotlib.pyplot as plt

      • import tensorflow as tf

      • import seaborn as sns

12. Designation of Internal and Public Functions¶

  • Internal Functions: Use a leading underscore (_) to indicate internal functions. Example: _internal_function.

  • Public Functions: No leading underscore, intended for the public API. Example: public_function.

13. Absolute Imports¶

  • Use absolute imports for clarity and robustness.

    • Do: .. code-block:

      from package_name.module import specific_function

    • Don’t: .. code-block:

      from .module import specific_function

14. Language Consistency¶

  • Code and Documentation: All code and documentation should be written in English to ensure clarity and accessibility for all team members and collaborators.

Adhering to these guidelines will ensure that the code created in the TDS team is clean, readable, and maintainable. If you have any questions or suggestions regarding these guidelines, please discuss them with the team.
